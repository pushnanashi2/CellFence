## Claims State

- Agent coordination claims are implemented as repository-local leases through
  `cellfence claim create`, `claim check`, and `claim list`.
- Claim stores are local JSON files, defaulting to `.cellfence/claims.json`.
  Local writes are serialized with a lock file and atomic rename.
- Claim enforcement covers active conflicts and unclaimed changed files.
- MCP exposes claim creation/checking as part of the agent-facing surface.
- Precision claim verification exists through `research:bundle` and
  `research:claim`, with protocol-driven thresholds and label validation.
- No broad "CellFence precision" claim is frozen at this baseline. External
  claims still require a reviewed corpus, evidence bundle, labels, and a passing
  claim report.

## Research State

- Corpus precision study exists for fixed exact-commit corpora and records
  clone, checkout, manifest, finding, and audit artifacts.
- Evidence bundle generation and validation exist, including normalized
  finding IDs, sampled findings, labels, and `SHA256SUMS`.
- History replay exists for before/after exact commits and introduced finding
  comparison.
- Upstream policy oracle v1 exists as mechanism validation for
  package-entry/dependency-hint ablation, not as independent architecture
  discovery or CellFence precision proof.
- Agent effectiveness A/B harness exists for local public-OSS task packs with
  `cellfence` and `control` arms, external run JSONL, adjudicated judgment
  JSONL, claim eligibility checks, and evidence hashing.
- Adversarial validation exists for language-independent, Python, and JS/TS
  scenarios with deterministic expected outcomes.
- Existing large public-OSS onboarding corpora are research inputs, not reviewed
  precision corpora.

## Conformance State

- Import conformance is the only standalone conformance suite.
- Current import suite covers ESM, CommonJS, dynamic import, `createRequire`,
  package subpaths, path aliases, file extensions, shadowed require controls,
  public imports, undeclared consumers, and computed-source diagnostics.
- Next standalone conformance suites should be added in this order:
  resolution, ownership, public surface, resources, ratchet, malformed inputs.
- Some of those behaviors are already covered by ordinary unit and CLI tests,
  but they are not yet frozen as named conformance ledgers.

## Boundaries For Next Work

- Do not weaken existing gates to pass refactors.
- Do not use baseline-only edits as a substitute for implementation fixes.
- Keep the public engine API stable while splitting `packages/engine/src/index.ts`.
- After each extraction or new conformance suite, run the targeted suite before
  the full gate set.
- Treat evidence graph and pure policy checker work as preparation for a future
  verifier; this baseline does not claim formal policy-conformance proof.
