# OSS Python 10 Onboarding Pilot, 2026-07-18

This pilot ran CellFence infer-manifest onboarding against ten public Python repositories at exact commits. It is a Python support and onboarding robustness check, not a precision claim.

Command:

```bash
npm run research:corpus --silent -- \
  --corpus docs/research/corpora/oss-python-10-2026-07-18.json \
  --out reports/corpus/oss-python-10-2026-07-18.json \
  --workdir tmp/corpus-python-10-2026-07-18 \
  --clone-mode shallow \
  --discard-checkouts \
  --infer-scope production
```

## Interpretation

The useful signal is that CellFence can now infer non-empty Python manifests from common Python packaging layouts: PEP 621 `pyproject.toml`, Poetry metadata, setuptools `setup.cfg`, static `setup.py` names/source roots, `src/<package>` layouts, and flat package layouts. The corpus completed without clone, manifest, configuration, parser, or timeout failures, and every inferred public entry was a Python file.

This does not prove that the 121 findings are upstream defects. The corpus uses `manifest.strategy: "infer"`, so findings measure detector pressure and onboarding friction until a human-reviewed manifest or upstream boundary oracle exists. The large `pytest` private-import cluster is a review target, not an automatic bug report.

The less flattering signal is also useful: unresolved Python imports remain visible in several mature repositories. Those are the next parser/root-resolution tuning targets, especially around optional imports, compatibility shims, generated modules, and package-specific import conventions.

## Limitations

- No target repository dependencies were installed.
- No target package scripts were run.
- No findings were manually labeled.
- Clean checks are not proof of architectural correctness.
- Findings are not proof of upstream bugs.
- The pilot does not measure recall or long-term developer value.
- Python framework resource adapters are still shallow compared with TypeScript/JavaScript resource coverage.
